# 微信公众号文章创作

## 定位

创作公众号文章内容，并为后续“文章配图”和“文章发布”节点准备清晰素材。默认输出可编辑 Markdown；当用户要求可直接复制到公众号编辑器时，再额外输出独立 `.html` 文件。

## 工作流

1. 理解选题：
   - 提炼用户目标、读者、场景、核心观点、发布目的和希望读者采取的行动。
   - 输入不足时先做合理假设，并在成稿前用一句话标注关键假设。
   - 保留用户给出的事实、数据、案例和品牌表述；不要编造来源、采访、引用或统计数字。
2. 建立内容质量基线：
   - 动笔前读取 `references/quality/content-quality.md`，用其中的检查项规避低创作度、搬运抄袭、内容空洞和低质 AIGC。
   - 每篇文章都要有明确的信息增量、原创判断、可信依据和读者可获得的阅读价值。
   - 使用 AI 可以辅助整理、改写和构思，但不要直接交付未经核验、未经人工判断和未经风格统一的生成文本或图片。
3. 匹配视觉风格：
   - 不按文章类型固定主题；把主题视作可自由选择的视觉风格，同一类文章可以使用任意主题。
   - 根据用户偏好、品牌气质、阅读节奏、信息密度、明暗程度、装饰强度和情绪温度选择风格。
   - 用户可以用 `--theme <主题>` 指定 HTML 主题，例如 `--theme warm-order`、`--theme 暖白秩序`、`--theme 青格笔记` 或 `--theme m2w-fullstack-blue`。指定后按该主题生成，不再自动改选。
   - 当用户要求公众号 HTML、视觉主题或视觉风格时，读取 `references/themes/index.md`，只加载一个最匹配的主题文件。
4. 设计文章结构：
   - 先给出标题候选、摘要/导语和大纲，再写正文。
   - 正文使用适合手机阅读的短段落、清晰小标题、必要的列表和自然过渡。
   - 开头要快速进入冲突、问题、趋势或收益；结尾要给出总结、行动建议或引导。
5. 写作与润色：
   - 默认使用中文公众号语气：清楚、克制、有观点，避免空泛鸡汤和夸张营销腔。
   - 可理解性以“聪明的十二岁学生能看懂”为底线：先解释术语，用具体例子承接抽象判断，句子短而清楚，复杂问题分层说明。
   - 解释型文章重视概念拆解、例子和可执行步骤；判断型文章重视框架、取舍和边界；观点型文章重视论证链条。
   - 对用户已有草稿，优先保留原观点，只调整结构、表达、节奏和标题。
   - 成稿后按 `references/quality/content-quality.md` 做一次快速自检；如缺少信息增量、证据或原创判断，先补强再输出。
6. 准备配图 prompt：
   - 为封面图输出一条可交给 `$wechat-article-image` 的 prompt，包含主题、核心隐喻、标题融合方式、比例、风格和负面约束。
   - 为关键章节规划 1-3 张文内配图 prompt，说明每张图对应的段落、视觉隐喻、比例和用途。
   - 长标题要压缩成适合画面的短主标题，完整标题仍保留在文章标题中。
7. 可选生成公众号 HTML：
   - 只有当用户要求“可复制到公众号编辑器”“排版 HTML”或“发布前 HTML”时，生成独立 `.html` 文件。
   - 使用简单结构和内联样式，避免外部 CSS、脚本、iframe、表单、事件处理器、远程字体和复杂选择器。
   - 默认不要生成正文 `<a>` 标签；外部资料统一收束到文末一个“参考资料”小节，保留标题和完整 URL。
   - 多个来源使用紧凑条目列表；不要为每条来源重复写“参考资料”“参考文章”等小标题。
   - 代码块不要依赖 `<pre>`、`<code>` 或 `white-space`。用带内联样式的段落/区块承载代码，换行写成 `<br>`，缩进和连续空格写成 `&nbsp;`。

## 配图 prompt 格式

为后续配图节点输出这个结构：

```text
用途：封面图 / 文内配图
文章标题：[完整标题]
画面标题：[适合画面内展示的短标题]
主题：[文章主题]
核心隐喻：[一句话说明画面如何解释观点]
构图：[封面用 2.35:1；文内图按段落需要选择 16:9 / 4:3 / 1:1]
标题融合：[屏幕 / 文档层 / 标签 / 面板 / 场景物体]
风格：[选定视觉风格和画面气质]
负面约束：无水印、无 Logo、无额外文字、无错别字、无随机 UI 标签
```

如果用户只要文章，不要配图方案，省略详细 prompt，只保留一行“建议配图方向”。

## 公众号 HTML 兼容规则

- 从浏览器复制渲染后的富文本，不要复制 HTML 源码。
- 如果用户提供 `--theme <主题>`，把它视为生成 HTML 的明确主题参数。参数值可以是主题文件 slug、主题中文名或不带 `theme-` 前缀的短名。
- `--theme` 可用值见 `references/themes/index.md` 的“主题参数”一节。无法识别时，不要猜测；先列出可用值并请用户确认。
- 样式保持内联且保守。
- 正文默认不要依赖 `<a>` 超链接，把外部资料做成文末一个“参考资料”小节。
- 多个来源使用同一个参考资料组连续列出；不要为每条来源重复生成“参考资料”“参考文章”等标签。
- 图片优先使用可发布的 HTTPS 地址。本地相对路径只适合本地预览。
- 不要在最终公众号图片地址里使用 `file://`、私有 IP、纯 `http`、非标准端口或本地相对路径。
- 图片标签保持简单：全宽、`height:auto`、可选小圆角、描述性的 `alt`。

## 输出

默认输出：

- 一篇可继续编辑的 Markdown 文章。
- 标题候选和推荐标题。
- 摘要/导语。
- 封面图 prompt 和必要的文内配图 prompt。

如果创建文件，除非用户指定其他位置，否则放在工作区 `outputs/` 目录。使用清晰文件名，例如：

```text
outputs/wechat-article.md
outputs/wechat-article.html
```

最终回复中链接产物文件，并说明下一步可使用 `$wechat-article-image` 生成图片，或使用 `$wechat-article-publish` 创建公众号草稿。

## 质量参考

- `references/quality/content-quality.md`：基于微信公众平台低创作度案例提炼的内容质量基线，用于规避同质化、搬运抄袭、低信息量和低质 AIGC。

## 主题参考

- `references/themes/index.md`：视觉风格选择指南，也是主题参数、主题文件和 HTML 用例的唯一索引。
- 当前主题库包含 40 个主题：10 个手写主题和 30 个由 markdown2wechat 转换而来的 `m2w-*` 主题。
- 手写主题适合优先推荐给普通文章生产，因为组件更克制、命名更稳定、适配当前写作流程更充分。
- `m2w-*` 主题用于需要更多色彩、社区风格、来源主题复刻或用户明确指定 markdown2wechat 风格时；参数示例：`m2w-fullstack-blue`、`theme-m2w-fullstack-blue`、`全栈蓝`。

主题文件定义颜色、排版、间距和可复用 HTML 片段。根据文章调整片段，而不是机械复制每个组件。选定主题后只读取对应的一个主题文件，不要一次性加载全部主题文档。

每个主题在 `references/themes/examples/` 下有一个可直接打开的 HTML 用例，用于预览该视觉风格的实际排版效果；入口页是 `references/themes/examples/index.html`。
