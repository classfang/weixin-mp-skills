# 微信公众号全流程编排

## 定位

作为流程控制层串联三个底层 skill：

- `$wechat-article-write`：负责选题理解、标题大纲、正文、视觉风格、配图 prompt 和可选公众号 HTML。
- `$wechat-article-image`：负责封面图、头图、文内配图和图片 prompt 优化。
- `$wechat-article-publish`：负责本地文章与图片上传、草稿创建、发布确认和状态查询。

不要把底层 skill 的细节复制到当前流程里。到达对应阶段时，读取并遵守相应 skill 的 `SKILL.md`；如果用户已经显式跳过某个阶段，只执行剩余阶段。

## 总原则

- 默认把目标推进到“可交付的本地文章 + 可用封面图 + 可创建公众号草稿”。只有用户明确要求并确认后，才正式发布。
- 用问答补齐关键缺口，但不要把用户拉进长表单。每轮最多问 3 个问题；能合理假设的问题先推进，并在阶段产物里标注假设。
- 对选择题给出推荐项，并说明差异。适合让用户选择的项包括目标读者、文章方向、标题风格、视觉主题、是否生成文内图、是否创建草稿、是否正式发布。
- 优先产出文件到工作区 `outputs/`。文件名要稳定清楚，例如 `outputs/wechat-article.md`、`outputs/wechat-article.html`、`outputs/wechat-cover.png`。
- 任何涉及真实微信 API 的动作前，先做 dry-run 或字段检查。任何正式发布动作前，必须获得当前轮次的明确确认。

## 快速决策

按用户输入选择入口：

- 只有选题或想法：进入“发现与定向”，然后完整执行写作、配图、草稿流程。
- 已有草稿、Markdown、HTML 或大纲：跳过从零构思，进入写作/改写阶段。
- 已有文章但缺图片：进入配图阶段，再询问是否创建草稿。
- 已有文章和封面图：进入发布准备阶段。
- 用户只想策划、不想生成文件：输出流程建议和下一步问题，不创建草稿。
- 用户说“全流程”“帮我发一篇”“从选题到发布”“不用我分别调用 skill”：使用本 skill 编排。

## 工作流

### 1. 发现与定向

先判断是否缺少会影响文章方向的核心信息。必要时询问：

- 主题或素材：选题、草稿、链接、要表达的观点或业务目标。
- 读者和目的：面向谁，读完希望相信什么、理解什么或采取什么行动。
- 约束：字数、语气、是否需要引用来源、是否要公众号 HTML、发布时间或品牌风格。

如果信息足以开始，直接给出一句“我将按这些假设推进”，然后进入写作。

### 2. 写作与排版

加载并使用 `$wechat-article-write`。

要求该阶段至少产出：

- 推荐标题和备选标题。
- 摘要/导语。
- 可继续编辑的 Markdown 正文。
- 适合公众号发布的 HTML，除非用户明确只要 Markdown。
- 封面图 prompt；如文章需要解释型视觉，再准备 1-3 条文内配图 prompt。

在写作阶段只问会明显改变成稿的问题，例如观点立场、目标读者、是否需要引用具体资料。标题、结构和局部措辞可以先给推荐方案。

### 3. 用户确认点

写作初稿完成后，给用户一个简短选择：

- 继续配图。
- 先调整标题/结构/语气。
- 只保存文章，不进入后续步骤。

如果用户在最初请求中已经明确要求全流程，可默认继续配图，同时说明可以随时回头改文章。

### 4. 配图

加载并使用 `$wechat-article-image`。

默认生成一张 2.35:1 公众号封面图。只有在文章确实需要图解、用户要求文内图，或正文已有明确文内配图 prompt 时，才继续生成文内图。

配图前可让用户选择视觉方向；如果用户没有偏好，采用写作阶段推荐的视觉风格。配图输出要记录：

- 图片路径或可复用 prompt。
- 用途：封面图、头图或文内配图。
- 比例和关键构图。
- 后续发布节点可使用的封面路径。

### 5. 发布准备

加载并使用 `$wechat-article-publish`。

创建草稿前检查：

- 文章文件路径存在，优先使用 HTML；没有 HTML 时用 Markdown。
- 标题明确，且正文中不会重复保留主标题。
- 封面图片路径或 `thumb_media_id` 已有。
- 如果正文有本地图片，确认图片文件存在且适合上传。
- 凭据来自 `~/.config/wechat-mp/.env`、环境变量或用户当轮提供；不要回显 AppSecret。

先运行发布节点的 dry-run 或等效字段检查。通过后，如果用户要求“创建草稿”或“全流程到草稿”，可以创建公众号草稿。最终回复要给出草稿结果和下一步发布确认方式。

### 6. 正式发布

只有用户在当前轮次明确说要发布，并且已知 `media_id` 或刚创建草稿成功，才进入正式发布。

执行发布前再次确认：

- 标题。
- 公众号草稿 `media_id`。
- 用户明确接受正式发布。

不得把“创建草稿”“准备发布”“可以发布”理解为正式发布确认。

## 问答策略

优先问少量高影响问题。可使用这些模板：

```text
我可以继续推进，但有 2 个选择会明显影响结果：
1. 文章方向：A 更偏方法论，B 更偏观点评论。我建议 A，因为更适合公众号收藏转发。
2. 视觉风格：A 冷静知识媒体，B 温暖编辑部。我建议 A，因为和主题的信息密度更匹配。
```

如果信息缺口不阻塞：

```text
我先按这些假设推进：读者是对这个话题有兴趣但不熟悉细节的普通公众号读者；目标是解释清楚并促成收藏。后面你可以随时让我改方向。
```

必须停止等待用户的情况：

- 用户需要提供原始素材、草稿文件或链接，否则无法写作。
- 用户需要选择多个互斥方向，且选错会导致整篇文章返工。
- 发布凭据缺失，且需要用户配置或提供。
- 准备正式发布，但尚未得到明确确认。

## 输出收束

每个阶段结束时只汇报最重要的产物和下一步：

- 写作完成：链接 Markdown/HTML，列出推荐标题和配图计划。
- 配图完成：展示或链接图片，说明封面路径。
- 草稿完成：给出草稿 `media_id` 或 API 返回摘要，提醒正式发布需要明确确认。
- 遇到配置错误：说明需要到 `https://developers.weixin.qq.com/platform` 核对 AppID/AppSecret、IP 白名单、接口权限或发布能力。

最终回复保持短而清楚，不要复述完整工作流。
