# 微信公众号全流程编排

## 定位

作为流程控制层串联三个底层 skill：

- `$wechat-article-write`：负责选题理解、标题大纲、正文、视觉风格、配图 prompt 和可选公众号 HTML。
- `$wechat-article-image`：负责封面图、头图、文内配图和图片 prompt 优化。
- `$wechat-article-publish`：负责本地文章与图片上传、草稿创建、发布确认和状态查询。

不要把底层 skill 的细节复制到当前流程里。到达对应阶段时，读取并遵守相应 skill 的 `SKILL.md`；如果用户已经显式跳过某个阶段，只执行剩余阶段。

## 总原则

- 默认主线为“选题/素材收集 -> 内容创作（包括配图 prompt） -> 配图 -> 写作排版 -> 用户确认 -> 创建草稿”。只有用户明确要求并确认后，才处理正式发布。
- 用问答补齐关键缺口，但不要把用户拉进长表单。每轮最多问 3 个问题；能合理假设的问题先推进，并在阶段产物里标注假设。
- 对选择题给出推荐项，并说明差异。适合让用户选择的项包括目标读者、文章方向、标题风格、视觉主题、是否生成文内图、是否创建草稿、是否正式发布。
- 优先产出文件到工作区 `outputs/`。文件名要稳定清楚，例如 `outputs/wechat-article.md`、`outputs/wechat-article.html`、`outputs/wechat-cover.png`。
- 任何涉及真实微信 API 的动作前，先做 dry-run 或字段检查。任何正式发布动作前，必须获得当前轮次的明确确认。

## 快速决策

按用户输入选择入口：

- 只有选题或想法：进入“选题/素材收集”，然后完整执行内容创作、配图、排版、确认和草稿流程。
- 已有草稿、Markdown、HTML 或大纲：跳过从零构思，进入内容创作/改写阶段；如已含配图 prompt，可直接进入配图。
- 已有文章但缺图片：先确认或补齐配图 prompt，进入配图阶段，再做写作排版。
- 已有文章和封面图：进入写作排版和用户确认，再创建草稿。
- 用户只想策划、不想生成文件：输出流程建议和下一步问题，不创建草稿。
- 用户说“全流程”“帮我发一篇”“从选题到发布”“不用我分别调用 skill”：使用本 skill 编排。

## 工作流

### 1. 选题/素材收集

先判断是否缺少会影响文章方向的核心信息。必要时询问：

- 主题或素材：选题、草稿、链接、要表达的观点或业务目标。
- 读者和目的：面向谁，读完希望相信什么、理解什么或采取什么行动。
- 约束：字数、语气、是否需要引用来源、是否要公众号 HTML、发布时间或品牌风格。

如果信息足以开始，直接给出一句“我将按这些假设推进”，然后进入内容创作。

### 2. 内容创作（包括配图 prompt）

加载并使用 `$wechat-article-write`。

要求该阶段至少产出：

- 推荐标题和备选标题。
- 摘要/导语。
- 可继续编辑的 Markdown 正文初稿。
- 封面图 prompt；如文章需要解释型视觉，再准备 1-3 条文内配图 prompt。

在内容创作阶段只问会明显改变成稿的问题，例如观点立场、目标读者、是否需要引用具体资料。标题、结构和局部措辞可以先给推荐方案。不要在这一阶段把公众号 HTML 当作最终排版交付；排版应等配图完成后再做。

### 3. 配图

加载并使用 `$wechat-article-image`。

默认生成一张 2.35:1 公众号封面图。只有在文章确实需要图解、用户要求文内图，或正文已有明确文内配图 prompt 时，才继续生成文内图。

配图前可让用户选择视觉方向；如果用户没有偏好，采用内容创作阶段推荐的视觉风格。配图输出要记录：

- 图片路径或可复用 prompt。
- 用途：封面图、头图或文内配图。
- 比例和关键构图。
- 后续排版和发布节点可使用的封面路径。

配图完成后必须形成图片交接清单，至少包含：

```text
封面图：path=outputs/wechat-cover.png，进入发布封面，不默认进入发布正文。
文内图 1：path=outputs/wechat-inline-1.png，对应章节=...，插入位置=...，alt=...
```

如果没有生成文内图，只生成封面图，也要在交接清单里明确“正文无文内图；排版稿只在预览版展示封面，发布版正文不重复封面”。

### 4. 写作排版

再次加载并使用 `$wechat-article-write`。

把已生成的图片纳入文章成品：

- 把图片交接清单作为排版阶段输入，不要只在回复中提到图片路径。
- 根据封面图和文内图结果调整标题、摘要、图注、段落衔接和图片插入位置。
- 在 Markdown 成稿中插入所有文内图，使用 `![描述](图片路径)`；每张文内图应出现在对应章节附近。
- 在公众号 HTML 中插入所有文内图，使用简单 `<img src="图片路径" alt="描述" style="width:100%;height:auto;...">`。
- 如果创建 HTML 文件，优先生成两份：`outputs/wechat-article-preview.html` 用于本地检查，顶部展示封面并包含所有文内图；`outputs/wechat-article.html` 用于发布，正文不重复封面但保留所有文内图。
- 如果只有封面图、没有文内图，排版输出必须明确区分“预览稿含封面”和“发布稿正文无图但发布封面使用该图片”。

排版交付前做图片回填检查：

- 列出每张生成图片的用途、文件路径和在 Markdown/HTML 中的位置。
- 有文内图时，确认 Markdown 至少包含对应 `![](...)`，HTML 至少包含对应 `<img`。
- 如果文内图未进入排版稿，先修复排版文件，再进入用户确认。

### 5. 用户确认

排版完成后，给用户一个简短确认点：

- 创建公众号草稿。
- 先调整标题/结构/语气/图片位置。
- 只保存本地文章和图片，不创建草稿。

如果用户在最初请求中已经明确要求全流程到草稿，可默认继续创建草稿，但仍要汇报将使用的标题、文章文件和封面图路径。

### 6. 创建草稿

加载并使用 `$wechat-article-publish`。

创建草稿前检查：

- 文章文件路径存在，优先使用 HTML；没有 HTML 时用 Markdown。
- 标题明确，且正文中不会重复保留主标题。
- 封面图片路径或 `thumb_media_id` 已有。
- 如果正文有本地图片，确认图片文件存在且适合上传。
- 凭据来自 `~/.config/wechat-mp/.env`、环境变量或用户当轮提供；不要回显 AppSecret。

先运行发布节点的 dry-run 或等效字段检查。通过后，如果用户要求“创建草稿”或“全流程到草稿”，可以创建公众号草稿。最终回复要给出草稿结果和下一步发布确认方式。

### 7. 后续正式发布（非默认主线）

只有用户在当前轮次明确说要发布，并且已知 `media_id` 或刚创建草稿成功，才进入正式发布。

执行发布前再次确认：

- 标题。
- 公众号草稿 `media_id`。
- 用户明确接受正式发布。

不得把“创建草稿”“准备发布”“可以发布”理解为正式发布确认。正式发布不是本 skill 默认主线的一部分。

## 问答策略

优先问少量高影响问题。可使用这些模板：

```text
我可以继续推进，但有 2 个选择会明显影响结果：
1. 文章方向：A 更偏方法论，B 更偏观点评论。我建议 A，因为更适合公众号收藏转发。
2. 视觉风格：A 冷静知识媒体，B 温暖编辑部。我建议 A，因为和主题的信息密度更匹配。
```

如果信息缺口不阻塞：

```text
我先按这些假设推进：读者是对这个话题有兴趣但不熟悉细节的普通公众号读者；目标是解释清楚并促成收藏。后面你可以随时让我改方向。
```

必须停止等待用户的情况：

- 用户需要提供原始素材、草稿文件或链接，否则无法写作。
- 用户需要选择多个互斥方向，且选错会导致整篇文章返工。
- 发布凭据缺失，且需要用户配置或提供。
- 准备正式发布，但尚未得到明确确认。

## 输出收束

每个阶段结束时只汇报最重要的产物和下一步：

- 内容创作完成：链接或保存 Markdown 初稿，列出推荐标题和配图 prompt。
- 配图完成：展示或链接图片，给出图片交接清单。
- 排版完成：链接 Markdown/HTML，列出将用于创建草稿的标题、文章文件、封面图，以及每张文内图在排版稿中的位置。
- 草稿完成：给出草稿 `media_id` 或 API 返回摘要，提醒正式发布需要另行明确确认。
- 遇到配置错误：说明需要到 `https://developers.weixin.qq.com/platform` 核对 AppID/AppSecret、IP 白名单、接口权限或发布能力。

最终回复保持短而清楚，不要复述完整工作流。
